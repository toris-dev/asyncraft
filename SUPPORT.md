# Support

asyncraft is a small, community-maintained utility library. Support is provided through public project channels on a best-effort basis.

## What is supported

- **Supported versions**: latest release and `main` branch.
- **Supported environments**: Node.js 18+ and officially documented API usage in [README](./README.md).
- **Preferred support flow**: public, reproducible, and documented requests only.

## Contact channels

- **Usage questions, how-to, and design discussion**
  - GitHub Discussions: <https://github.com/toris-dev/asyncraft/discussions>
- **Bugs and feature requests**
  - GitHub Issue: <https://github.com/toris-dev/asyncraft/issues>
  - Bug issue template: [.github/ISSUE_TEMPLATE/bug_report.md](./.github/ISSUE_TEMPLATE/bug_report.md)
  - Feature issue template: [.github/ISSUE_TEMPLATE/feature_request.md](./.github/ISSUE_TEMPLATE/feature_request.md)
- **Security issues**
  - Private advisory path only: <https://github.com/toris-dev/asyncraft/security/advisories/new>
  - Or email: `ironjustlikethat@gmail.com` (non-public details only)

## Support policy

- We focus on:
  - Fixes and regressions that affect documented public APIs.
  - Clear bug reports with reproducible steps.
  - Feature requests with concrete API and behavior proposals.
- We do not provide:
  - Private/custom implementation support.
  - Immediate guarantees on response time or fix ETA.
  - Full production support contracts.
- We may close issues without comment when they are duplicates, stale, or out of project scope.

## Bug report baseline

Before filing a bug report, please include:

- asyncraft version and Node.js version.
- Minimal reproduction snippet or test.
- Expected vs. actual behavior.
- Error output / logs if available.

These requirements are mirrored in the [bug report template](./.github/ISSUE_TEMPLATE/bug_report.md).

## Feature request baseline

Before filing a feature request, please include:

- The problem to solve and affected user story.
- Proposed API shape or usage example.
- Alternative approaches considered.
- Possible behavioral impact (new API, behavior change, docs-only).

These requirements are mirrored in the [feature request template](./.github/ISSUE_TEMPLATE/feature_request.md).

## Escalation

For unresolved concerns after issue triage, use the issue thread and ping maintainers in the same thread with reproducible evidence. Do not create duplicate reports.
